Transformer fine-tuning은 사전학습된 모델의 일반 지식을 특정 업무, 도메인, 출력 형식에 맞게 조정하는 과정입니다. 중요한 점은 "모델을 추가로 학습한다"가 아니라 "모델이 어떤 상황에서 어떤 답을 해야 하는지 더 분명하게 만든다"에 있습니다.

이 글은 fine-tuning을 처음 검토할 때 바로 학습 코드로 들어가기 전에 확인해야 할 판단 기준을 정리합니다.

## Fine-tuning이 필요한 순간

먼저 확인할 것은 현재 문제가 프롬프트 개선이나 검색 증강만으로 풀리는지입니다. 아래 조건에 가까울수록 fine-tuning의 가치가 커집니다.

- 반복적으로 같은 형식의 출력을 만들어야 한다.
- 조직 내부의 표현 방식, 분류 체계, 작성 톤을 안정적으로 맞춰야 한다.
- 프롬프트에 예시를 많이 넣어도 결과가 흔들린다.
- 짧은 지연 시간과 낮은 토큰 비용이 중요하다.
- 모델이 특정 도메인의 판단 기준을 일관되게 따라야 한다.

반대로 최신 사실 조회, 긴 문서 근거 인용, 계속 바뀌는 사내 지식 반영이 핵심이라면 fine-tuning보다 검색 증강 구조가 먼저일 가능성이 높습니다.

## 시작 전에 고정할 세 가지

### 1. 목적

Fine-tuning 목적은 가능한 한 관찰 가능한 문장으로 적어야 합니다. "답변 품질 향상"은 목표가 너무 넓습니다. 대신 "고객 문의를 12개 내부 카테고리 중 하나로 분류한다", "기술 문서를 우리 팀 릴리즈 노트 톤으로 5문장 이내 요약한다"처럼 입력, 출력, 성공 조건이 드러나야 합니다.

### 2. 데이터

좋은 fine-tuning 데이터는 양보다 일관성이 먼저입니다. 같은 입력에 대해 서로 다른 라벨이나 문체가 섞여 있으면 모델은 평균적인 답을 학습합니다. 초기에는 적은 수의 샘플이라도 사람이 보기에 "이 답이 맞다"라고 설명할 수 있는 예제를 모으는 편이 낫습니다.

데이터를 준비할 때는 다음을 확인합니다.

- 입력과 출력 형식이 실제 서비스 상황과 같은가
- 애매한 케이스가 별도 규칙으로 설명되어 있는가
- 실패 예시와 경계 사례가 포함되어 있는가
- 학습 데이터와 평가 데이터가 중복되지 않는가
- 민감 정보나 불필요한 개인정보가 제거되어 있는가

### 3. 평가 기준

Fine-tuning은 학습 손실이 낮아졌다고 바로 성공이 아닙니다. 실제 사용자가 보는 품질 기준으로 평가해야 합니다. 분류 문제라면 정확도, F1, 혼동 행렬을 볼 수 있고, 생성 문제라면 형식 준수율, 금지 표현 위반율, 사람이 보는 선호 평가가 더 중요할 수 있습니다.

평가 기준은 학습 전에 정해야 합니다. 그래야 모델을 바꿨을 때 좋아진 것인지, 단지 느낌이 다른 것인지 구분할 수 있습니다.

## Full fine-tuning과 PEFT

모든 파라미터를 다시 학습하는 full fine-tuning은 제어력이 크지만 비용과 운영 부담도 큽니다. 반면 LoRA 같은 parameter-efficient fine-tuning 방식은 기존 모델 대부분을 고정하고 작은 어댑터만 학습합니다. 초기 실험에서는 PEFT 계열 방식이 비용, 반복 속도, 롤백 편의성 면에서 유리한 경우가 많습니다.

선택 기준은 단순합니다.

- 데이터가 작고 실험을 빠르게 돌려야 한다면 PEFT부터 검토한다.
- 모델의 핵심 동작을 크게 바꿔야 하고 충분한 데이터와 인프라가 있다면 full fine-tuning을 검토한다.
- 운영에서 여러 도메인 버전을 나눠야 한다면 어댑터 기반 구조가 관리하기 쉽다.

## 운영에서 놓치기 쉬운 부분

Fine-tuning 모델은 배포 후에도 계속 관찰해야 합니다. 데이터 분포가 바뀌면 성능이 떨어질 수 있고, 사용자가 새로운 입력 패턴을 만들면 기존 평가셋이 현실을 대표하지 못할 수 있습니다.

운영 체크리스트는 다음과 같습니다.

- 배포 전 기준 모델과 fine-tuned 모델을 같은 평가셋으로 비교한다.
- 실패 사례를 저장하고 다음 데이터 개선 루프에 반영한다.
- 모델 버전, 학습 데이터 버전, 평가 결과를 함께 기록한다.
- 비용, 지연 시간, 안전성 기준을 모델 품질과 같이 본다.
- 문제가 생겼을 때 이전 모델로 되돌릴 수 있는 절차를 둔다.

## 결론

Transformer fine-tuning은 강력하지만, 시작점은 학습 코드가 아니라 문제 정의입니다. 목적, 데이터, 평가 기준이 선명하면 작은 실험으로도 충분히 배울 수 있습니다. 반대로 이 세 가지가 흐리면 더 큰 모델과 더 긴 학습도 문제를 해결하지 못합니다.

처음 시도한다면 작게 시작하는 편이 좋습니다. 하나의 명확한 업무, 검증 가능한 평가셋, 비교 가능한 기준 모델을 정하고 개선 폭을 확인한 뒤 다음 범위로 넓히는 접근이 가장 현실적입니다.
